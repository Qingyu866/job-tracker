## 重构动机

### 原始问题

在原始实现中，Agent 方法返回 **JSON 字符串**，调用方需要手动解析：

```java
// ❌ 原始实现（返回 JSON 字符串）
public interface ViceInterviewerAgent {
    String decideNextStep(String context);  // 返回 JSON 字符串
}

public interface ExpertEvaluatorAgent {
    String evaluate(String context);  // 返回 JSON 字符串
}

// 调用方需要手动解析
String jsonResult = agents.viceInterviewer().decideNextStep(context);
NextStepDecision decision = objectMapper.readValue(jsonResult, NextStepDecision.class);
```

**问题**：
1. ❌ **类型不安全**：JSON 字符串容易出错，编译期无法检查
2. ❌ **代码冗余**：每个调用方都需要解析 JSON
3. ❌ **维护困难**：JSON 结构变化需要修改多处代码
4. ❌ **错误处理复杂**：需要处理 JSON 解析异常

---

### LangChain4j 的解决方案

LangChain4j **原生支持**对象类型的自动序列化/反序列化：

```java
// ✅ 重构后（返回类型化对象）
public interface ViceInterviewerAgent {
    NextStepDecision decideNextStep(String context);  // 直接返回对象
}

public interface ExpertEvaluatorAgent {
    EvaluationResult evaluate(String context);  // 直接返回对象
}

// 调用方直接使用对象
NextStepDecision decision = agents.viceInterviewer().decideNextStep(context);
// 无需手动解析，LangChain4j 自动处理
```

**优势**：
1. ✅ **类型安全**：编译期检查，IDE 自动补全
2. ✅ **代码简洁**：无需手动解析 JSON
3. ✅ **易于维护**：结构变化只需修改 DTO 类
4. ✅ **错误处理简单**：LangChain4j 自动处理序列化异常

## 重构效果对比

### 调用方代码对比

#### 原始实现（返回 JSON 字符串）

```java
// ❌ 调用方需要手动解析
String jsonResult = agents.viceInterviewer().decideNextStep(context);

try {
    ObjectMapper mapper = new ObjectMapper();
    NextStepDecision decision = mapper.readValue(jsonResult, NextStepDecision.class);

    // 使用 decision
    if (decision.getAction() == Action.NEXT_QUESTION) {

} catch (JsonProcessingException e) {
```

```java
// ✅ 直接使用对象，无需解析
NextStepDecision decision = agents.viceInterviewer().decideNextStep(context);

if (decision.getAction() == NextStepDecision.Action.NEXT_QUESTION) {
```

## LangChain4j 工作原理

### 自动序列化/反序列化流程

```
┌─────────────────────────────────────────────────────────────┐
│                        调用流程                              │
└─────────────────────────────────────────────────────────────┘

1. Java 调用
   NextStepDecision decision = agent.decideNextStep(context);

                ↓

2. LangChain4j 拦截
   - 检测到返回类型是 NextStepDecision
   - 自动在 System Message 中添加 JSON Schema

                ↓

3. LLM 生成 JSON
   {
     "action": "NEXT_QUESTION",
     "nextTopic": "Redis 集群",
     ...
   }

                ↓

4. LangChain4j 反序列化
   - 将 JSON 字符串转换为 NextStepDecision 对象
   - 自动处理枚举类型（Action, TopicSource 等）
   - 如果解析失败，抛出异常

                ↓

5. 返回对象
   NextStepDecision decision = ...  // 已完成反序列化
```

---

### 关键点

1. **DTO 类要求**
   - ✅ 必须有无参构造函数（`@NoArgsConstructor`）
   - ✅ 必须有 Getter/Setter（`@Data`）
   - ✅ 枚举类型会被自动处理

2. **LangChain4j 自动处理**
   - ✅ 自动生成 JSON Schema
   - ✅ 自动反序列化 JSON 为对象
   - ✅ 自动处理嵌套对象和集合

3. **异常处理**
   - ✅ LangChain4j 会在反序列化失败时抛出异常
   - ✅ 异常信息会指出哪个字段解析失败

## 注意事项

### ⚠️ DTO 类设计原则

1. **必须有无参构造函数**
   ```java
   @NoArgsConstructor  // 必须有
   public class NextStepDecision { ... }
   ```

2. **必须有 Getter/Setter**
   ```java
   @Data  // Lombok 自动生成
   public class NextStepDecision { ... }
   ```

3. **枚举类型使用标准 Java 枚举**
   ```java
   public enum Action {
       NEXT_QUESTION,
       FINISH_INTERVIEW
   }
   ```

4. **嵌套对象同样适用**
   ```java
   public class EvaluationResult {
       private ScoreDetail scores;  // 嵌套对象
       private CredibilityAssessment credibilityAssessment;  // 嵌套对象
   }
   ```

---

### ⚠️ System Message 调整

1. **移除 JSON 格式说明**
   - LangChain4j 会自动生成 JSON Schema
   - 不需要在 System Message 中说明 JSON 格式

2. **使用字段名而非 JSON 键名**
   ```java
   // ✅ 正确（使用字段名）
   - action: 动作类型（NEXT_QUESTION, FINISH_INTERVIEW）

   // ❌ 错误（使用 JSON 键名）
   - "action": 动作类型
   ```

3. **枚举值使用大写**
   ```java
   // ✅ 正确
   topicSource: PROJECT_DEEP_DIVE

   // ❌ 错误
   topicSource: "PROJECT_DEEP_DIVE"
   ```

---

### ⚠️ 异常处理

虽然 LangChain4j 会自动处理反序列化，但仍需处理可能的异常：

```java
try {
    NextStepDecision decision = agents.viceInterviewer().decideNextStep(context);
    // 使用 decision
} catch (Exception e) {
    log.error("Agent 调用失败", e);
    // 降级处理
    return NextStepDecision.builder()
        .action(Action.FINISH_INTERVIEW)
        .reason("Agent 调用失败，结束面试")
        .build();
}
```

## 总结

   - DTO 类集中管理数据结构

4. **框架集成**

   - 遵循框架最佳实践
