# ID 生成策略优化和 Long 精度问题修复

**日期**: 2026-03-18
**版本**: v2.0.0

---

## 📋 问题描述

### 问题 1: ID 生成策略使用数据库自增

**现象**:
- 大部分实体类使用 `IdType.AUTO` (数据库自增)
- 导致分布式环境下可能出现 ID 冲突
- 不利于系统扩展和分布式部署

### 问题 2: Long 类型在 JSON 传递时精度丢失

**错误日志**:
```
参数类型不匹配：参数 'resumeId' 类型不匹配，期望类型：Long
```

**原因**:
- JavaScript 的 Number 类型最大安全整数是 `2^53 - 1` (即 `9007199254740991`)
- 数据库自增 ID 和雪花算法生成的 ID 可能超出这个范围
- 后端 Long 类型序列化为 JSON 后，前端接收时精度丢失

---

## 🔧 解决方案

### 1. 修改 MyBatis Plus 全局 ID 生成策略

**文件**: `src/main/resources/application.yml`

**修改前**:
```yaml
mybatis-plus:
  global-config:
    db-config:
      id-type: auto  # 数据库自增
```

**修改后**:
```yaml
mybatis-plus:
  global-config:
    db-config:
      id-type: assign_id  # 雪花算法
```

---

### 2. 配置 Jackson 序列化 Long 为 String

**文件**: `src/main/java/com/jobtracker/config/JacksonConfig.java`

**添加的配置**:
```java
// 配置 Long 类型序列化为 String，避免前端 JavaScript 精度丢失
// JavaScript 的 Number 类型最大安全整数是 2^53 - 1，雪花算法生成的 Long 类型会超出这个范围
javaTimeModule.addSerializer(Long.class, ToStringSerializer.instance);
javaTimeModule.addSerializer(Long.TYPE, ToStringSerializer.instance);
```

**原理**:
- 使用 Jackson 的 `ToStringSerializer` 将所有 Long 类型序列化为字符串
- 前端接收到字符串类型的 ID，不会出现精度丢失
- 前端在传递 ID 给后端时，后端会自动将字符串反序列化为 Long

## 📊 性能影响

### 雪花算法特点

- ✅ **高性能**: 本地生成，无需访问数据库
- ✅ **分布式**: 全局唯一，支持多节点部署
- ✅ **趋势递增**: 按时间递增，有利于数据库索引
- ✅ **无单点**: 不依赖数据库，避免单点故障

### 序列化性能影响

- ⚠️ **轻微增加**: 字符串比数字多占用少量内存
- ✅ **可接受**: 相比精度丢失带来的问题，性能影响可忽略

---

## 🔄 数据库迁移说明

### 现有数据处理

**重要**: 现有数据库中的自增 ID 无需修改，可以继续使用。

**原因**:
- 雪花算法生成的 ID 范围: `[0, ~9223372036854775807]`
- 数据库自增 ID 范围: `[1, ~18446744073709551615]` (取决于 BIGINT)
- 两者不会冲突，可以共存

**新数据生成策略**:
- 新创建的实体使用雪花算法生成 ID
- ID 值通常大于现有自增 ID

## ⚠️ 注意事项

### 1. 前端无需修改

**原因**:
- 后端自动将 Long 序列化为字符串
- 前端接收到的是字符串类型
- 前端发送请求时，后端自动反序列化为 Long

**示例**:

```typescript
// 前端代码（无需修改）
const response = await axios.get('/api/applications/1234567890123456789');
// 后端接收到: 1234567890123456789 (Long 类型)
```

### 2. 数据库表结构无需修改

**原因**:
- ID 字段类型已经是 `BIGINT`
- 雪花算法生成的 ID 可以存储在 BIGINT 中

## 📚 参考资料

### MyBatis Plus ID 生成策略

- `IdType.AUTO`: 数据库自增
- `IdType.ASSIGN_ID`: 雪花算法（默认）
- `IdType.ASSIGN_UUID`: UUID
- `IdType.INPUT`: 手动输入

### JavaScript Number 精度问题

- 最大安全整数: `Number.MAX_SAFE_INTEGER` = `9007199254740991`
- 超出此范围的整数会丢失精度
- 解决方案: 使用字符串传递大整数
