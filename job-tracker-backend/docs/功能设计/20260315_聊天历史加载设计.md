# 聊天历史加载设计

**创建日期**：2026-03-15
**作者**：Job Tracker Team
**状态**：待审批

---

## 📋 需求背景

### 问题描述
当前系统实现了多会话隔离的聊天记忆功能，使用 `SafeTurnBasedChatMemoryProvider` 在 JVM 内存中存储对话历史。虽然消息会持久化到数据库，但服务重启后，内存中的 ChatMemory 不会自动从数据库恢复历史消息。

### 影响
- 服务重启后，AI 无法"记住"之前的对话
- 用户需要重新开始对话，体验不连贯
- 历史上下文丢失，AI 可能给出不一致的回答

### 需求目标
在会话加载时，自动从数据库拉取最近的 N 条消息，恢复到 JVM 内存中的 ChatMemory，并确保消息序列的正确性。

---

## 🎯 设计方案

### 核心思路
在 `SafeTurnBasedChatMemoryProvider.get()` 创建新的 ChatMemory 实例时：
1. 从数据库加载该会话的历史消息
2. 转换为 LangChain4j ChatMessage 格式
3. **验证并修复消息序列**（关键步骤）
4. 添加到 ChatMemory 中

### 架构图

```
┌─────────────────────────────────────────────────────────────────┐
│                     服务启动/会话加载时                            │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│  1. 前端连接 WebSocket → ChatWebSocketHandler                    │
│                          ↓                                      │
│  2. 调用 JobAgentFactory.createAgent(sessionKey)                │
│                          ↓                                      │
│  3. 调用 chatMemoryProvider.get(sessionKey)                     │
│                          ↓                                      │
│  4. 判断是否是新创建的 ChatMemory                               │
│                          ↓                                      │
│     ┌──────────────┬──────────────────────┐                    │
│     │   新创建      │      已缓存            │                    │
│     ↓              ↓                       ↓                    │
│  ┌──────────────────────────────────────────────────────────┐  │
│  │  5. 从数据库加载最近 N 条消息                               │  │
│  │  6. 转换为 LangChain4j ChatMessage                         │  │
│  │  7. 验证并修复消息序列（三层修复）                          │  │
│  │     - 修复开头：删除无效的 ASSISTANT/TOOL                  │  │
│  │     - 修复中间：删除连续的同类型消息                        │  │
│  │     - 修复末尾：删除未完成的 USER 消息                      │  │
│  └──────────────────────────────────────────────────────────┘  │
│                          ↓                                      │
│  8. 返回已恢复历史的 ChatMemory                                 │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```

```java
/**
 * 查询会话最近的 N 条消息（按时间升序）
 *
 * @param sessionId 会话ID
 * @param limit     限制条数
 * @return 消息列表（按时间升序）
 */
default List<ChatMessage> selectRecentMessages(Long sessionId, int limit) {
    // 先倒序查询最近的 N 条，然后反转成正序
    List<ChatMessage> messages = selectList(new LambdaQueryWrapper<ChatMessage>()
            .eq(ChatMessage::getSessionId, sessionId)
            .orderByDesc(ChatMessage::getCreatedAt)
            .last("LIMIT " + limit));

    // 反转成升序
    Collections.reverse(messages);
    return messages;
}
```

### 4. 修改 SafeTurnBasedChatMemoryProvider

**文件**：`src/main/java/com/jobtracker/agent/memory/SafeTurnBasedChatMemoryProvider.java`

```java
@Slf4j
public class SafeTurnBasedChatMemoryProvider implements ChatMemoryProvider {

    private final int maxMessages;
    private final ConcurrentHashMap<Object, SafeTurnBasedChatMemory> memoryStore = new ConcurrentHashMap<>();

    // 新增：注入 ChatHistoryService
    private final ChatHistoryService chatHistoryService;
    private final int historyLoadLimit;  // 加载历史消息的数量

    public SafeTurnBasedChatMemoryProvider(
            int maxMessages,
            ChatHistoryService chatHistoryService,
            int historyLoadLimit) {
        this.maxMessages = maxMessages;
        this.chatHistoryService = chatHistoryService;
        this.historyLoadLimit = historyLoadLimit;
    }

    @Override
    public ChatMemory get(Object memoryId) {
        // computeIfAbsent：如果不存在则创建，存在则返回缓存的
        return memoryStore.computeIfAbsent(memoryId, id -> {
            log.info("创建新的 ChatMemory：memoryId={}", id);

            // 1. 创建空的 ChatMemory
            SafeTurnBasedChatMemory memory = new SafeTurnBasedChatMemory(id, maxMessages);

            // 2. 从数据库加载历史消息
            if (chatHistoryService != null && id instanceof String) {
                String sessionKey = (String) id;
                loadHistoryFromDatabase(memory, sessionKey);
            }

            return memory;
        });
    }

    /**
     * 从数据库加载历史消息到 ChatMemory
     */
    private void loadHistoryFromDatabase(SafeTurnBasedChatMemory memory, String sessionKey) {
        try {
            // 查询最近的消息
            List<com.jobtracker.entity.ChatMessage> dbMessages =
                chatHistoryService.getRecentMessages(sessionKey, historyLoadLimit);

            if (dbMessages.isEmpty()) {
                log.debug("没有历史消息需要加载：sessionKey={}", sessionKey);
                return;
            }

            // 转换并添加到 ChatMemory
            for (com.jobtracker.entity.ChatMessage dbMsg : dbMessages) {
                dev.langchain4j.data.message.ChatMessage chatMessage = convertToChatMessage(dbMsg);
                memory.add(chatMessage);
            }

            log.info("已加载历史消息：sessionKey={}, count={}", sessionKey, dbMessages.size());

            // ⚠️ 关键：验证并修复消息序列
            ensureMessageOrderCorrect(memory);

        } catch (Exception e) {
            log.error("加载历史消息失败：sessionKey={}", sessionKey, e);
            // 失败不影响会话创建，只是没有历史而已
        }
    }

    /**
     * 修复消息顺序，避免 "Conversation roles must alternate" 错误
     * <p>
     * 修复策略：
     * 1. SYSTEM 锚定：若存在 SYSTEM，以其为起点，之前的全部丢弃（解决 USER 在 SYSTEM 前的问题）
     * 2. 严格交替：SYSTEM 后 USER/ASSISTANT 必须交替，SYSTEM 本身重置交替状态
     * 3. 结尾完整：删除末尾的 USER（避免未完成的请求发送给模型）
     * </p>
     */
    private void ensureMessageOrderCorrect(SafeTurnBasedChatMemory memory) {
        List<ChatMessage> messages = memory.getMessages();

        if (messages == null || messages.isEmpty()) {
            return;
        }

        int originalSize = messages.size();
        List<ChatMessage> fixed = new ArrayList<>(originalSize);

        // 1. 【关键改进】预扫描：找到第一个 SYSTEM 的位置
        // 如果存在 SYSTEM，它必须是有效对话的起点，之前的全部视为脏数据
        int firstSystemIndex = -1;
        for (int i = 0; i < messages.size(); i++) {
            if (messages.get(i).type() == ChatMessageType.SYSTEM) {
                firstSystemIndex = i;
                break;
            }
        }

        // 确定遍历的起始位置
        int startIndex = (firstSystemIndex != -1) ? firstSystemIndex : 0;

        // 状态变量
        boolean firstUserFound = false;   // 仅在无 SYSTEM 模式下需要
        ChatMessageType lastType = null;  // 上一条非 SYSTEM 消息的类型
        int lastUserIndex = -1;           // fixed 列表中最后一条 USER 的索引

        // 2. 从确定的起点开始遍历
        for (int i = startIndex; i < messages.size(); i++) {
            ChatMessage msg = messages.get(i);
            ChatMessageType type = msg.type();

            // --- 处理 SYSTEM 消息 ---
            if (type == ChatMessageType.SYSTEM) {
                fixed.add(msg);
                lastType = null;  // 🔑 关键修复：SYSTEM 作为分隔符，重置交替状态
                // 注意：如果是第一个 SYSTEM (i==startIndex)，它自然成为列表第一个
                // 如果是中间的 SYSTEM，它也会重置 lastType，允许后续接 USER 或 ASSISTANT
                continue;
            }

            // --- 处理无 SYSTEM 模式下的开头 ---
            // 只有在没有 SYSTEM 消息时，才需要寻找第一条 USER 作为起点
            if (firstSystemIndex == -1 && !firstUserFound) {
                if (type == ChatMessageType.USER) {
                    firstUserFound = true;
                    fixed.add(msg);
                    lastType = type;
                    lastUserIndex = fixed.size() - 1;
                }
                // 跳过开头的 ASSISTANT/TOOL
                continue;
            }

            // --- 处理严格交替 (USER/ASSISTANT) ---
            // 此时 firstUserFound 必然为 true (要么有 SYSTEM，要么已找到 USER)
            if (type == lastType) {
                // 发现连续同类型 (如 USER->USER)，跳过以修复交替错误
                continue;
            }

            // 添加有效消息
            fixed.add(msg);
            lastType = type;

            if (type == ChatMessageType.USER) {
                lastUserIndex = fixed.size() - 1;
            }
        }

        // 3. 【结尾修复】确保不以 USER 结尾
        // 如果最后一条是 USER，说明用户发完消息还没得到回答，发送给模型会报错
        if (lastUserIndex >= 0 && lastUserIndex == fixed.size() - 1) {
            fixed.remove(lastUserIndex);
        }

        // 4. 更新内存
        messages.clear();
        messages.addAll(fixed);

        int fixedSize = messages.size();
        if (originalSize != fixedSize) {
            log.info("消息序列修复：memoryId={}, 原始={}, 修复={}, 移除={}",
                     memory.id(), originalSize, fixedSize, originalSize - fixedSize);
        }
    }

    /**
     * 将数据库消息转换为 LangChain4j ChatMessage
     */
    private dev.langchain4j.data.message.ChatMessage convertToChatMessage(
            com.jobtracker.entity.ChatMessage dbMsg) {

        String role = dbMsg.getRole();
        String content = dbMsg.getContent();

        return switch (role.toUpperCase()) {
            case "USER" -> dev.langchain4j.data.message.UserMessage.from(content);
            case "ASSISTANT" -> dev.langchain4j.data.message.AiMessage.from(content);
            case "SYSTEM" -> dev.langchain4j.data.message.SystemMessage.from(content);
            default -> throw new IllegalArgumentException("Unknown role: " + role);
        };
    }
}
```

## 📊 消息序列验证

### 正确的消息序列

```
[SYSTEM*] (USER ASSISTANT)+
```
- `SYSTEM*` = 0 条或多条 SYSTEM 消息
- `(USER ASSISTANT)+` = 至少一对完整的对话轮次
- **SYSTEM 作为分隔符**：重置交替状态，允许后续接 USER 或 ASSISTANT
- 不允许连续的同类型消息（除 SYSTEM）
- 必须以 ASSISTANT 结束（如果有对话内容）

### 算法关键改进

| 特性 | 说明 |
|------|------|
| **SYSTEM 锚定** | 预扫描找到第一个 SYSTEM，以其为起点，之前的全部丢弃 |
| **SYSTEM 分隔符** | `lastType = null` - SYSTEM 重置交替状态 |
| **严格交替** | USER/ASSISTANT 必须交替出现 |

### 问题场景示例

#### 场景 1：SYSTEM 锚定（USER 在 SYSTEM 前）
```
修复前: [USER0, ASSISTANT0, SYSTEM1, USER1, ASSISTANT1]
                             ↓ 以第一个 SYSTEM 为起点
修复后: [SYSTEM1, USER1, ASSISTANT1]
```
**说明**：USER0 和 ASSISTANT0 在第一个 SYSTEM 前，被丢弃

#### 场景 2：中间连续 USER
```
修复前: [USER1, ASSISTANT1, USER2, USER3, ASSISTANT2]
                              ↓ 跳过 USER2
修复后: [USER1, ASSISTANT1, USER3, ASSISTANT2]
```

#### 场景 3：中间连续 ASSISTANT
```
修复前: [USER1, ASSISTANT1, ASSISTANT2, USER2]
                              ↓ 跳过 ASSISTANT1
修复后: [USER1, ASSISTANT2, USER2]
```

#### 场景 4：末尾 USER
```
修复前: [USER1, ASSISTANT1, USER2]
                              ↓ 删除末尾 USER2
修复后: [USER1, ASSISTANT1]
```

#### 场景 5：开头异常（无 SYSTEM）
```
修复前: [ASSISTANT0, USER1, ASSISTANT1, USER2, ASSISTANT2]
         ↓ 跳过开头的 ASSISTANT0
修复后: [USER1, ASSISTANT1, USER2, ASSISTANT2]
```

#### 场景 6：SYSTEM 作为分隔符
```
修复前: [USER1, ASSISTANT1, SYSTEM2, ASSISTANT3, USER3]
                                        ↓ SYSTEM 重置，ASSISTANT3 被跳过
修复后: [SYSTEM2, USER3]
```
**说明**：SYSTEM 重置 `lastType`，ASSISTANT3 与 SYSTEM 不连续，但作为对话开头是无效的，被跳过

#### 场景 7：综合修复
```
修复前: [USER0, ASSISTANT0, SYSTEM1, USER1, ASSISTANT1, USER2, USER3, ASSISTANT2, USER4]
                             ↓ 以 SYSTEM1 为起点
中间:   [SYSTEM1, USER1, ASSISTANT1, USER2, USER3, ASSISTANT2, USER4]
                                        ↓ 跳过 USER2
中间:   [SYSTEM1, USER1, ASSISTANT1, USER3, ASSISTANT2, USER4]
                                                            ↓ 删除末尾 USER4
修复后: [SYSTEM1, USER1, ASSISTANT1, USER3, ASSISTANT2]
```

## 🤔 设计考虑

### 1. 为什么只加载最近 N 条？
- **Token 成本**：加载全部历史会消耗大量 Token
- **性能考虑**：大量消息加载和处理耗时
- **实际需求**：用户通常只需要最近的上下文

### 2. 加载多少条合适？
- **默认 10 条**：约 5 轮对话（用户 + AI）
- **可配置**：通过 `history-load-limit` 调整
- **建议范围**：5-20 条

### 3. 为什么需要三层修复？

| 修复层次 | 问题来源 | 影响 |
|---------|---------|------|
| 开头 | 数据损坏、异常写入 | 消息序列不完整 |
| 中间 | 并发写入、服务崩溃 | 连续同类型消息 |
| 末尾 | 服务重启（AI 未回复） | 后续消息被误删 |

## ✅ 验收标准

### 基础功能
- [ ] 服务重启后，使用相同 sessionKey 连接
- [ ] ChatMemory 自动从数据库加载最近的消息
- [ ] AI 能够理解之前的对话上下文
- [ ] 加载数量可配置
- [ ] 新会话（无历史）正常工作
- [ ] 加载失败不影响会话创建

### 消息序列验证
- [ ] **开头检查**：删除开头的 ASSISTANT/TOOL 消息（保留 SYSTEM）
- [ ] **中间检查**：删除连续的同类型消息（USER-USER 或 ASSISTANT-ASSISTANT）
- [ ] **末尾检查**：删除末尾的 USER 消息，确保以 ASSISTANT 结束
- [ ] **序列正确性**：修复后的序列必须是 SYSTEM* → (USER ASSISTANT)+
