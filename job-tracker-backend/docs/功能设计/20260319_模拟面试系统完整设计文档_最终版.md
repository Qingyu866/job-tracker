## 零、LangChain4j 自动 JSON 序列化机制

### 0.1 核心特性

**LangChain4j 框架自动处理 JSON 序列化/反序列化**：

```java
// ✅ 正确做法：直接返回 Java 对象
public interface ViceInterviewerAgent {
    List<QuestionPlanDTO> generateQuestionPlan(String context);
}

// ❌ 错误做法：返回 String 然后手动解析
public interface ViceInterviewerAgent {
    String generateQuestionPlan(String context);  // 不要这样做！
}
```

**工作原理**：
1. 调用 Agent 方法时，LangChain4j 会自动将参数序列化为 JSON
2. AI 返回 JSON 格式的响应
3. LangChain4j 会自动将 JSON 反序列化为指定的 Java 类型
4. **完全不需要手动处理 JSON**

---

### 0.2 正确的 Agent 方法定义

**示例 1：返回列表**

```java
public interface ViceInterviewerAgent {
    /**
     * 生成考察计划
     *
     * @return 考察计划列表（LangChain4j 自动解析）
     */
    List<QuestionPlanDTO> generateQuestionPlan(String context);
}
```

**示例 2：返回单个对象**

```java
public interface ExpertEvaluatorAgent {
    /**
     * 评估用户回答
     *
     * @return 评估结果（LangChain4j 自动解析）
     */
    EvaluationResult evaluate(String context);
}
```

**示例 3：返回自定义对象**

```java
public interface SkillGeneratorAgent {
    /**
     * 生成技能标签
     *
     * @return 技能标签列表（LangChain4j 自动解析）
     */
    List<SkillTag> generateSkillTags(List<String> skillNames);
}
```

---

### 0.3 DTO 定义要求

**LangChain4j 能正确解析的 DTO 必须满足**：

1. ✅ **有无参构造函数**
   ```java
   @NoArgsConstructor
   public class QuestionPlanDTO { }
   ```

2. ✅ **有 getter/setter 或使用 Lombok**
   ```java
   @Data  // 自动生成 getter/setter
   public class QuestionPlanDTO { }
   ```

3. ✅ **字段名称与 JSON 匹配**
   - Java: `roundNumber`
   - JSON: `{"round_number": 1}`
   - LangChain4j 会自动转换驼峰命名

4. ✅ **支持嵌套对象**
   ```java
   @Data
   public class EvaluationResult {
       private Scores scores;  // 嵌套对象
       private String feedback;
   }
   ```

---

### 0.4 System Message 中的格式说明

**❌ 不要这样做**：
```
# ⚠️ 输出格式要求
严禁使用 Markdown 代码块！
直接输出纯 JSON 数组
```

**✅ 应该这样做**：
```
# 输出格式
返回 JSON 数组，每个元素包含：
- round_number: 轮次
- skill_name: 技能名称
...

# 输出示例
[
  {
    "round_number": 1,
    "skill_name": "Redis"
  }
]
```

**原因**：
- LangChain4j 会自动清理 Markdown 代码块（```json...```）
- 不需要在 System Message 中强调"不要使用 Markdown"
- 保持提示词简洁自然即可

---

### 0.5 错误处理

**如果 AI 返回的不是有效 JSON**：

LangChain4j 会抛出异常，我们可以这样处理：

```java
try {
    List<QuestionPlanDTO> plans = agents.viceInterviewer().generateQuestionPlan(context);
    return plans;
} catch (Exception e) {
    log.error("AI 生成考察计划失败", e);
    // 返回默认计划或抛出业务异常
    throw new BusinessException("生成考察计划失败，请重试");
}
```

## 一、核心需求

### 1.1 用户核心建议

**建议 1：预生成考察计划**
> "既然这个面试的问题是由岗位的 JD 和用户的简历来生成的，那么要考察的知识点是不是就可以让副面试官先根据这个内容和面试的轮次来生成这个要考察的知识点和对应难度"

**核心思想**：
1. **在创建会话时**，副面试官生成完整的 25 轮考察计划
2. **考察计划包含**：每轮考察什么技能、难度等级、问题类型
3. **按照计划执行**，而不是每轮都动态决策
4. **持久化到数据库**，支持暂停/恢复

---

**建议 2：动态补全技能标签**
> "agent已经知道他当前考察的知识点是什么那么他就可以，先去将他这一次要考察的知识点的名称转为list去查询数据库看数据库了里面缺少哪些，将缺少的在返回给ai让其生成这个skill_tags"

**核心思想**：
1. AI 评估时，已经知道当前考察的技能名称
2. **在评估前**，先查询数据库 `skill_tags`，看缺少哪些
3. 将缺少的技能列表返回给 AI
4. 让 AI 生成新的 `skill_tags` 记录（批量插入）

---

**建议 3：确保简历摘要字段**
> "确保其至少要有这个 user_resumes 表里面的这个 summary 字段，因为确实有人没有工作经历和这个拿得出收的项目嘛"

**核心思想**：
1. 有些人没有工作经历和项目经验（应届生、转行者）
2. **必须确保 `user_resumes.summary` 字段不为空**
3. 如果用户没有填写，使用 AI 生成默认摘要

---

**建议 4：考察计划持久化**
> "还要将这个副面试官生成的要考察的知识点和已经考察完来的知识点持久化到数据库里面，这样用在后续暂停对话然后又回来的时候副面试管就知道当前考察的知识点的进度"

**核心思想**：
1. 考察计划必须持久化到数据库
2. 记录每个知识点的考察状态（待考察/已完成/跳过）
3. 支持暂停后恢复，副面试官知道当前进度
4. 避免重复考察或遗漏

---

### 1.2 当前实现的问题

**问题 1：缺少考察计划**
- ❌ 副面试官不知道应该考察什么
- ❌ 没有明确的考察计划
- ❌ 无法确保 JD 要求的技能都被覆盖

**问题 2：所有评估都是"通用"**
- ❌ 数据库记录显示 `skillId=0, skillName="通用"`
- ❌ 新技能没有对应的 `skill_tags` 记录
- ❌ 用户体验差，看不懂具体考察了什么

**问题 3：无法暂停/恢复**
- ❌ 面试中途中断后无法继续
- ❌ 用户需要一次性完成 25 轮
- ❌ 体验不友好

## 二、三个 Agent 职责边界

### 2.1 MainInterviewerAgent（主面试官）

**角色定位**: 面试的主导者，负责与候选人直接对话

**核心职责**:
1. **提问**: 根据考察计划，生成具体的问题
2. **欢迎**: 在面试开始时，进行欢迎和开场
3. **总结**: 在面试结束时，对整体表现进行总结

**不应该做的事**:
- ❌ 决定下一个话题（由考察计划决定）
- ❌ 评估候选人回答（这是评审专家的职责）
- ❌ 分析简历差距（这是评审专家的职责）

**能力边界**:
- ✅ 擅长：基于具体话题生成自然、专业的问题
- ✅ 擅长：根据候选人的回答进行追问
- ✅ 擅长：营造良好的面试氛围

---

### 2.2 ViceInterviewerAgent（副面试官）

**角色定位**: 面试的大脑，负责生成考察计划

**核心职责**:
1. **生成考察计划**: 在创建会话时，基于 JD 和简历生成完整的 25 轮考察计划
2. **改进建议**: 在面试结束时，提供针对性的改进建议

**不应该做的事**:
- ❌ 直接向候选人提问（这是主面试官的职责）
- ❌ 评估回答质量（这是评审专家的职责）
- ❌ 每轮动态决策（改为在创建会话时一次性生成计划）

**能力边界**:
- ✅ 擅长：策略性思考，平衡 JD 要求 vs 简历内容
- ✅ 擅长：规划完整的考察路径
- ✅ 擅长：判断面试进度，控制节奏

---

### 2.3 ExpertEvaluatorAgent（评审专家）

**角色定位**: 面试的观察者，负责评估和分析

**核心职责**:
1. **评估**: 评估候选人的回答质量（技术准确性、逻辑清晰度、深度）
2. **差距分析**: 对比简历声称与实际表现，识别夸大成分
3. **可信度评分**: 计算简历可信度评分（0-1）
4. **技能分析**: 生成每个技能的可信度分析报告

**不应该做的事**:
- ❌ 决定下一个话题（由考察计划决定）
- ❌ 直接提问（这是主面试官的职责）
- ❌ 提供改进建议（这是副面试官的职责）

**能力边界**:
- ✅ 擅长：客观评估回答质量
- ✅ 擅长：识别技术深度和广度
- ✅ 擅长：对比简历声称与实际表现

## 三、表结构设计（合并表方案）

### 3.1 设计原则

**核心思想**：避免表冗余，将考察计划和评分记录合并到 `mock_interview_evaluations` 表

**类比**：
- **考察计划** = 考试大纲（考试前就知道要考什么）
- **评分记录** = 试卷评分（考试后才有的分数）
- **合并表** = 一张表包含计划和结果

---

```sql
CREATE TABLE mock_interview_evaluations (
  evaluation_id BIGINT PRIMARY KEY AUTO_INCREMENT,
  session_id VARCHAR(50) NOT NULL,
  round_number INT NOT NULL,

  -- 技能信息
  skill_id BIGINT,
  skill_name VARCHAR(100) NOT NULL,

  -- 考察计划字段（新增）⭐
  plan_status VARCHAR(20) DEFAULT 'PENDING' COMMENT '计划状态：PENDING(待考察), IN_PROGRESS(执行中), COMPLETED(已完成), SKIPPED(跳过), PAUSED(暂停中)',
  topic_source VARCHAR(50) COMMENT '选题来源',
  question_type VARCHAR(50) COMMENT '问题类型',
  planned_difficulty INT COMMENT '计划难度（1-5）',
  actual_difficulty INT COMMENT '实际难度',
  context_info TEXT COMMENT '上下文信息',
  reason TEXT COMMENT '选择原因',

  -- 面试执行字段
  question_text TEXT COMMENT '问题内容',
  user_answer TEXT COMMENT '用户回答',

  -- 评分字段
  technical_score DECIMAL(4,1) COMMENT '技术准确性（0-4）',
  logic_score DECIMAL(3,1) COMMENT '逻辑清晰度（0-3）',
  depth_score DECIMAL(3,1) COMMENT '深度与广度（0-3）',
  total_score DECIMAL(4,1) COMMENT '总分（0-10）',
  feedback TEXT COMMENT '详细反馈',
  suggestion TEXT COMMENT '改进建议',
  keywords_matched VARCHAR(500) COMMENT '匹配的关键词',
  keywords_missing VARCHAR(500) COMMENT '缺失的关键词',

  -- 时间戳
  evaluated_at TIMESTAMP,
  created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP,
  updated_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP,

  -- 索引
  INDEX idx_session_status (session_id, plan_status),
  INDEX idx_round_skill (round_number, skill_name),
  INDEX idx_session_round (session_id, round_number)
) COMMENT='模拟面试评分记录表（包含考察计划）';
```

| `planned_difficulty` | INT | 计划难度（1-5） |

| `context_info` | TEXT | 上下文（如："简历声称精通 Redis"） |

## 四、考察计划预生成机制

### 4.1 生成时机

**时机**：`MockInterviewService.createSession()`

**流程**：
```
创建会话
    ↓
解析 JD 和简历
    ↓
调用副面试官生成考察计划（25 轮）
    ↓
确保所有技能存在（动态补全）
    ↓
批量插入 mock_interview_evaluations（plan_status = 'PENDING'）
    ↓
更新会话信息（total_plans, current_plan_id）
    ↓
创建会话成功
```

---

### 4.2 ViceInterviewerAgent 新增方法

**方法签名**：
```java
/**
 * 生成完整的考察计划（25轮）
 *
 * @param context 上下文信息（包含 JD 和简历数据）
 * @return 考察计划列表
 */
List<QuestionPlanDTO> generateQuestionPlan(String context);
```

### 4.3 QuestionPlanDTO 定义

```java
     */
    private String topicSource;

    /**
     * 问题类型：PROJECT_DEEP_DIVE, SKILL_VERIFICATION, JD_REQUIREMENT, GENERAL, OPEN_ENDED
     */
    private String questionType;

    /**
     * 难度等级（1-5）
     */
    private Integer difficulty;

    /**
     * 上下文信息（例如：简历声称精通Redis）
     */
    private String contextInfo;

    /**
     * 选择该技能的原因
     */
    private String reason;
}
```

### 4.4 考察策略

**副面试官生成计划的策略**：

```
第 1 轮：
- 项目深挖（建立对话）
- 验证 1 个简历声称的技能

第 2-10 轮：
- JD 核心技能验证
- 优先验证简历中声称熟练掌握的技能

第 11-20 轮：
- 技能深度考察
- 补充 JD 要求但简历未提及的技能

第 21-25 轮：
- 通用能力考察（算法、数据结构、系统设计）
- 或 HR 问题
```

## 五、动态技能补全机制

### 5.1 问题分析

**当前流程**：
```
AI 评估返回 "Redis" → 查询 skill_tags → 如果不存在，返回 0（通用）→ 用户体验差
```

**数据库记录**：
```
skill_id | skill_name | category
---------|-----------|----------
1        | Java      | 编程语言
2        | Redis     | 数据库
```

**问题**：
- ❌ AI 评估提到 "Elasticsearch"，但数据库中没有
- ❌ 只能使用默认值 0（通用）
- ❌ 所有新技能都显示为"通用"，用户看不懂

---

### 5.2 解决方案：ensureSkillsExist()

**时机**：创建会话时（生成考察计划后）

**流程**：
```
副面试官生成考察计划（包含技能名称）
    ↓
提取所有技能名称：["Redis", "Spring Boot", "Elasticsearch", ...]
    ↓
查询数据库，找出缺失的技能：["Elasticsearch", ...]
    ↓
调用 AI 生成缺失的技能标签
    ↓
批量插入 skill_tags 表
    ↓
返回技能名称 → ID 的映射
    ↓
保存考察计划时使用真实的 skillId
```

---

### 5.3 代码实现

#### 5.3.1 确保技能存在

```java
/**
 * 确保技能标签存在于数据库中
 * <p>
 * 如果技能不存在，使用 AI 生成技能信息并批量插入
 * </p>
 *
 * @param skillNames 技能名称列表
 * @return 技能名称 -> 技能ID 的映射
 */
private Map<String, Long> ensureSkillsExist(List<String> skillNames) {
    Map<String, Long> skillMap = new HashMap<>();

    // 1. 查询已存在的技能
    List<SkillTag> existingSkills = skillTagMapper.selectList(
        new LambdaQueryWrapper<SkillTag>()
            .in(SkillTag::getSkillName, skillNames)
    );

    // 2. 构建已存在技能的映射
    for (SkillTag skill : existingSkills) {
        skillMap.put(skill.getSkillName(), skill.getSkillId());
    }

    // 3. 找出缺失的技能
    List<String> missingSkills = skillNames.stream()
        .filter(name -> !skillMap.containsKey(name))
        .collect(Collectors.toList());

    // 4. 如果没有缺失的，直接返回
    if (missingSkills.isEmpty()) {
        return skillMap;
    }

    // 5. 使用 AI 生成缺失的技能信息 ⭐
    log.info("检测到 {} 个缺失的技能，使用 AI 生成：{}", missingSkills.size(), missingSkills);

    // LangChain4j 会自动将 AI 返回的 JSON 解析为 List<SkillTag>
    List<SkillTag> newSkills = agents.skillGenerator().generateSkillTags(missingSkills);

    // 6. 批量插入新技能
    for (SkillTag skill : newSkills) {
        skillTagMapper.insert(skill);
        skillMap.put(skill.getSkillName(), skill.getSkillId());
        log.info("创建新技能：{} (ID: {})", skill.getSkillName(), skill.getSkillId());
    }

    return skillMap;
}
```

## 六、暂停/恢复机制

### 6.1 典型场景

**场景 1：面试中途中断**
```
第 5 轮 → 用户有事离开 → 点击"暂停" → 2天后回来 → 点击"继续" → 从第 6 轮继续
```

**场景 2：分次完成面试**
```
第 1 天：完成第 1-5 轮 → 暂停
第 2 天：继续第 6-10 轮 → 暂停
...
第 5 天：完成第 21-25 轮 → 结束
```
