## 📚 知识点概述

在开发基于 AI 的聊天应用时，一个常见但容易被忽视的问题是：**聊天历史的持久化和消息序列的正确性**。

本知识点文档记录了：
- 聊天历史持久化的通用方案
- "Conversation roles must alternate" 错误的本质和解决
- 消息序列验证算法的设计思路
- 最佳实践和可复用代码模式

---

## 🔍 问题本质

### 问题 1：服务重启后记忆丢失

**典型症状**：
```
用户: "我叫张三"
AI: "好的张三，有什么可以帮你的？"

[服务重启]

用户: "我多大了？"
AI: "抱歉，我不知道你的信息。" ← 记忆丢失！
```

**根因**：
- 聊天历史只存储在 JVM 内存中
- 虽然消息已持久化到数据库，但 ChatMemory 不会自动恢复

### 问题 2：消息序列错误导致模型调用失败

**典型症状**：
```
Exception: Conversation roles must alternate
```

**根因**：
- LangChain4j 等框架要求消息角色必须交替（USER → ASSISTANT → USER → ASSISTANT）
- 数据库中的消息可能因为各种原因不满足这个规则：
  - 服务崩溃时 AI 还没回复（末尾是 USER）
  - 并发写入导致连续同类型消息
  - 数据损坏导致开头有 ASSISTANT
  - SYSTEM 消息位置异常

## 💡 通用解决方案

### 方案 1：聊天历史持久化架构

```
┌─────────────────────────────────────────────────────────────────┐
│                      会话加载时机                                │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│  1. 新会话开始 / 服务重启后重连                                  │
│                          ↓                                      │
│  2. ChatMemoryProvider.get(memoryId)                           │
│                          ↓                                      │
│  3. 判断是新创建还是已缓存                                      │
│                          ↓                                      │
│  4. 【新创建】执行以下步骤：                                     │
│     a. 从数据库查询最近 N 条消息                                │
│     b. 转换为框架格式的 ChatMessage                             │
│     c. 验证并修复消息序列                                        │
│     d. 添加到 ChatMemory                                        │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```

### 方案 2：消息序列验证算法

#### 正确的消息序列规则

```
[SYSTEM*] (USER ASSISTANT)+
```

| 规则 | 说明 |
|------|------|
| `SYSTEM*` | 0 条或多条 SYSTEM 消息 |
| `(USER ASSISTANT)+` | 至少一对完整的对话轮次 |
| **SYSTEM 作为分隔符** | SYSTEM 重置交替状态，允许后续接任意角色 |
| **严格交替** | USER/ASSISTANT 必须交替出现（除 SYSTEM） |
| **完整结尾** | 必须以 ASSISTANT 结束（如果有对话内容） |

#### 三层修复策略

| 修复层次 | 问题场景 | 修复方法 |
|---------|---------|----------|
| **开头** | 开头有 ASSISTANT/TOOL | 跳过，找到第一条有效消息 |
| **开头** | SYSTEM 前有 USER/ASSISTANT | 以第一个 SYSTEM 为起点，丢弃之前所有 |
| **中间** | 连续的同类型消息 | 跳过重复的消息 |
| **末尾** | 末尾是 USER 消息 | 删除（对话未完成） |

## 🔧 可复用代码模板

### 模板 1：数据库查询最近消息

```java
/**
 * 查询会话最近的 N 条消息（按时间升序）
 *
 * @param sessionId 会话ID
 * @param limit     限制条数
 * @return 消息列表（按时间升序）
 */
default List<ChatMessage> selectRecentMessages(Long sessionId, int limit) {
    // 先倒序查询最近的 N 条
    List<ChatMessage> messages = selectList(new LambdaQueryWrapper<ChatMessage>()
            .eq(ChatMessage::getSessionId, sessionId)
            .orderByDesc(ChatMessage::getCreatedAt)
            .last("LIMIT " + limit));

    // 反转成升序（确保按时间顺序处理）
    Collections.reverse(messages);
    return messages;
}
```

### 模板 2：消息序列验证和修复

```java
/**
 * 修复消息顺序，避免 "Conversation roles must alternate" 错误
 * <p>
 * 修复策略：
 * 1. SYSTEM 锚定：若存在 SYSTEM，以其为起点，之前的全部丢弃
 * 2. 严格交替：SYSTEM 后 USER/ASSISTANT 必须交替，SYSTEM 重置交替状态
 * 3. 结尾完整：删除末尾的 USER（避免未完成的请求发送给模型）
 * </p>
 *
 * @param messages 消息列表（会被直接修改）
 */
public static void ensureMessageOrderCorrect(List<ChatMessage> messages) {
    if (messages == null || messages.isEmpty()) {
        return;
    }

    int originalSize = messages.size();
    List<ChatMessage> fixed = new ArrayList<>(originalSize);

    // 1. 预扫描：找到第一个 SYSTEM 的位置
    int firstSystemIndex = -1;
    for (int i = 0; i < messages.size(); i++) {
        if (messages.get(i).type() == ChatMessageType.SYSTEM) {
            firstSystemIndex = i;
            break;
        }
    }

    // 2. 确定遍历起点
    int startIndex = (firstSystemIndex != -1) ? firstSystemIndex : 0;

    // 3. 状态变量
    boolean firstUserFound = false;   // 仅在无 SYSTEM 模式下需要
    ChatMessageType lastType = null;  // 上一条非 SYSTEM 消息的类型
    int lastUserIndex = -1;           // fixed 中最后一条 USER 的索引

    // 4. 遍历并修复
    for (int i = startIndex; i < messages.size(); i++) {
        ChatMessage msg = messages.get(i);
        ChatMessageType type = msg.type();

        // --- 处理 SYSTEM 消息 ---
        if (type == ChatMessageType.SYSTEM) {
            fixed.add(msg);
            lastType = null;  // 🔑 SYSTEM 作为分隔符，重置交替状态
            continue;
        }

        // --- 处理无 SYSTEM 模式下的开头 ---
        if (firstSystemIndex == -1 && !firstUserFound) {
            if (type == ChatMessageType.USER) {
                firstUserFound = true;
                fixed.add(msg);
                lastType = type;
                lastUserIndex = fixed.size() - 1;
            }
            // 跳过开头的 ASSISTANT/TOOL
            continue;
        }

        // --- 处理严格交替 ---
        if (type == lastType) {
            // 跳过连续的同类型消息
            continue;
        }

        // 添加有效消息
        fixed.add(msg);
        lastType = type;

        if (type == ChatMessageType.USER) {
            lastUserIndex = fixed.size() - 1;
        }
    }

    // 5. 删除末尾的 USER（对话未完成）
    if (lastUserIndex >= 0 && lastUserIndex == fixed.size() - 1) {
        fixed.remove(lastUserIndex);
    }

    // 6. 更新原列表
    messages.clear();
    messages.addAll(fixed);
}
```

### 模板 3：集成到 ChatMemoryProvider

```java
@Slf4j
public class PersistentChatMemoryProvider implements ChatMemoryProvider {

    private final ChatHistoryService historyService;
    private final int historyLoadLimit;
    private final ConcurrentHashMap<Object, ChatMemory> cache = new ConcurrentHashMap<>();

    @Override
    public ChatMemory get(Object memoryId) {
        return cache.computeIfAbsent(memoryId, id -> {
            // 1. 创建空的 ChatMemory
            ChatMemory memory = new ChatMemory();

            // 2. 从数据库加载历史
            if (historyService != null && id instanceof String) {
                List<ChatMessage> history = historyService.getRecentMessages(
                    (String) id, historyLoadLimit);

                // 3. 转换并添加
                for (ChatMessage dbMsg : history) {
                    memory.add(toChatMessage(dbMsg));
                }

                // 4. 验证并修复消息序列
                List<ChatMessage> messages = memory.getMessages();
                MessageOrderValidator.ensureMessageOrderCorrect(messages);

                log.info("加载历史消息：id={}, count={}", id, history.size());
            }

            return memory;
        });
    }

    private ChatMessage toChatMessage(ChatMessage dbMsg) {
        return switch (dbMsg.getRole().toUpperCase()) {
            case "USER" -> UserMessage.from(dbMsg.getContent());
            case "ASSISTANT" -> AiMessage.from(dbMsg.getContent());
            case "SYSTEM" -> SystemMessage.from(dbMsg.getContent());
            default -> throw new IllegalArgumentException("Unknown role");
        };
    }
}
```

## ⚠️ 常见陷阱

### 陷阱 1：忘记反转消息列表

**错误示例**：
```java
// ❌ 错误：直接使用倒序结果
List<ChatMessage> messages = selectList(
    new LambdaQueryWrapper<ChatMessage>()
        .orderByDesc(ChatMessage::getCreatedAt)  // 倒序
        .last("LIMIT 10")
);
// 直接添加到 ChatMemory，导致时间顺序错误！
```

**正确做法**：
```java
// ✅ 正确：先倒序查询，再反转成升序
List<ChatMessage> messages = selectList(...);  // 倒序
Collections.reverse(messages);  // 反转成升序
```

### 陷阱 2：没有验证消息序列

**风险**：
```
数据库: [USER1, ASSISTANT1, USER2]  ← 末尾是 USER
         ↓ 直接加载
ChatMemory: [USER1, ASSISTANT1, USER2]
         ↓ 用户发新消息 USER3
ChatMemory: [USER1, ASSISTANT1, USER2, USER3]  ← 连续 USER！
         ↓ 发送给模型
Exception: Conversation roles must alternate
```

**正确做法**：
```java
// 加载后必须验证和修复
ensureMessageOrderCorrect(messages);
```

### 陷阱 3：SYSTEM 消息处理不当

**错误理解**：SYSTEM 消息和普通消息一样，不能连续

**正确理解**：SYSTEM 消息可以作为分隔符，重置交替状态

```
✅ 正确：
[SYSTEM1, USER1, ASSISTANT1, SYSTEM2, USER2, ASSISTANT2]
        ↑                          ↑
     重置状态                    再次重置

❌ 错误：
[USER1, SYSTEM1, ASSISTANT1]
  ↑ 普通消息无法重置
```

### 陷阱 4：加载太多历史消息

**问题**：
- Token 成本高
- 性能影响大
- 可能超出模型上下文限制

**建议**：
```java
// 默认加载 10 条（约 5 轮对话）
int historyLoadLimit = 10;

// 根据模型上下文窗口调整
// GPT-3.5: 10-20 条
// GPT-4: 20-50 条
// Claude: 20-100 条
```

## 🎯 最佳实践

### 1. 懒加载策略

```java
// ✅ 推荐：只在新创建时加载
return cache.computeIfAbsent(memoryId, id -> {
    // 只有第一次创建时才从数据库加载
    return loadFromDatabase(id);
});

// ❌ 不推荐：每次都加载
public ChatMemory get(Object memoryId) {
    if (cache.containsKey(memoryId)) {
        return cache.get(memoryId);
    }
    return loadFromDatabase(memoryId);  // 每次都查数据库
}
```

### 2. 优雅的错误处理

```java
// ✅ 推荐：加载失败不影响会话创建
try {
    loadHistoryFromDatabase(memory, sessionKey);
} catch (Exception e) {
    log.error("加载历史消息失败：sessionKey={}", sessionKey, e);
    // 失败不影响会话创建，只是没有历史而已
}

// ❌ 不推荐：加载失败直接抛异常
public ChatMemory get(Object memoryId) {
    List<ChatMessage> history = loadFromDatabase(memoryId);  // 可能抛异常
    // ...
}
```

### 3. 配置化加载参数

```yaml
# application.yml
chat:
  memory:
    max-messages: 20          # ChatMemory 最大消息数
    history-load-limit: 10    # 从数据库加载的历史数量
```

```java
@Value("${chat.memory.history-load-limit:10}")
private int historyLoadLimit;
```

### 4. 监控和日志

```java
// 记录加载和修复情况
if (originalSize != fixedSize) {
    log.info("消息序列修复：id={}, 原始={}, 修复={}, 移除={}",
             id, originalSize, fixedSize, originalSize - fixedSize);
}
```
