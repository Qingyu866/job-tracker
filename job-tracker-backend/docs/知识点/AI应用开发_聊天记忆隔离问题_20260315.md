# AI 应用开发：聊天记忆隔离问题

## 📚 知识点概述

在开发基于 LangChain4j（或其他 AI 框架）的应用时，一个常见但容易被忽视的问题是：**多个用户/会话之间聊天记忆互相干扰**。

本知识点文档记录了：
- 问题的本质和原因
- 通用的解决方案
- 最佳实践和避坑指南
- 可复用的代码模式

---

## 🔍 问题本质

### 什么是聊天记忆隔离问题？

**定义**：不同用户或不同会话之间的对话历史互相串扰，导致 AI 能够"记住"其他会话的内容。

### 典型症状

```
用户 A（sessionId=aaa）: "我叫张三"
AI: "好的张三，有什么可以帮你的？"

用户 B（sessionId=bbb）: "你好"
AI: "你好张三！" ← 错误！不应该知道用户 A 的信息
```

### 业务影响

- **隐私泄露**：用户可以看到其他用户的对话内容
- **用户体验差**：AI 混淆不同会话的上下文
- **数据安全风险**：敏感信息可能跨会话泄露

---

## 🔧 技术根因

### 架构层面的原因

在大多数 AI 应用框架（如 LangChain4j）中，AI Agent 通常被设计为**单例 Bean**：

```java
@Configuration
public class AIConfig {
    @Bean
    public AIWorker aiWorker(ChatMemoryProvider provider) {
        return AiServices.builder(AIWorker.class)
            .chatMemoryProvider(provider)  // ← Provider 虽然支持多记忆
            .build();                       // ← 但 Agent 是单例！
    }
}
```

**问题链条**：
1. `ChatMemoryProvider` 可以根据 `memoryId` 创建不同的记忆
2. 但 `AIWorker.chat()` 方法通常**不接收 `memoryId` 参数**
3. 结果：所有请求都使用同一个默认的记忆实例

### 为什么框架这样设计？

1. **简化使用**：大多数简单应用只需要一个全局 Agent
2. **性能考虑**：单例避免重复创建 Agent
3. **历史包袱**：早期设计没有考虑多租户场景

## 💡 通用解决方案

### 方案选择矩阵

| 方案 | 复杂度 | 隔离性 | 性能 | 适用场景 |
|------|--------|--------|------|----------|
| **动态 Agent + 工厂模式** | 中 | ⭐⭐⭐⭐⭐ | 中 | **推荐**，多用户应用 |
| 单例 Agent + ThreadLocal | 低 | ⭐⭐⭐ | 高 | 单机单用户 |
| 原型 Agent（每次创建） | 低 | ⭐⭐⭐⭐⭐ | 低 | 低并发场景 |
| 修改框架源码 | 高 | ⭐⭐⭐⭐⭐ | 高 | 需要深度定制 |

### 推荐方案：动态 Agent + 工厂模式

#### 架构图

```
┌─────────────────────────────────────────────────────────┐
│                   WebSocket/HTTP 请求                    │
│                   sessionId: "user-123"                  │
└──────────────────────┬──────────────────────────────────┘
                       │
                       ▼
┌─────────────────────────────────────────────────────────┐
│              ChatWebSocketHandler / Controller           │
│                                                         │
│   String sessionId = extractSessionId();               │
│   AIWorker agent = aiWorkerFactory.createAgent(sessionId);│
│   String response = agent.chat(userMessage);           │
└──────────────────────┬──────────────────────────────────┘
                       │
                       ▼
┌─────────────────────────────────────────────────────────┐
│                 AIWorkerFactory                         │
│                                                         │
│   public AIWorker createAgent(String sessionId) {       │
│       ChatMemory memory = provider.get(sessionId);     │
│       return AiServices.builder(AIWorker.class)         │
│           .chatMemory(memory)  // ← 每次传入独立记忆     │

└──────────────────────┬──────────────────────────────────┘
                       │
                       ▼
┌─────────────────────────────────────────────────────────┐
│           ChatMemoryProvider (内部缓存)                  │
│                                                         │
│   Map<Object, ChatMemory> memoryStore = new ...();      │
│                                                         │
│   public ChatMemory get(Object memoryId) {             │
```

#### 代码模板

```java
// 1. AI 工厂
@Component
@RequiredArgsConstructor
public class AIWorkerFactory {

    private final ChatModel chatModel;
    private final ChatMemoryProvider memoryProvider;
    private final List<Object> tools; // 工具列表

    public AIWorker createWorker(String sessionId) {
        // 从 Provider 获取或创建记忆（内部有缓存）
        ChatMemory memory = memoryProvider.get(sessionId);

        return AiServices.builder(AIWorker.class)
            .chatModel(chatModel)
            .chatMemory(memory)  // 关键：每次传入独立的 memory
            .tools(tools)
            .build();
    }
}

// 2. 在 Handler/Controller 中使用
@RestController
@RequiredArgsConstructor
public class ChatController {

    private final AIWorkerFactory aiWorkerFactory;

    @PostMapping("/chat")
    public String chat(@RequestBody ChatRequest request) {
        String sessionId = request.getSessionId();

        // 动态创建 Agent
        AIWorker worker = aiWorkerFactory.createWorker(sessionId);

        // 调用
        return worker.chat(request.getMessage());
    }
}
```

## ⚠️ 常见陷阱

### 陷阱 1：重复缓存

**错误示例**：

```java
// ❌ 错误：ChatMemoryProvider 内部已经有缓存了
@Service
public class MemoryManager {
    private final Map<String, ChatMemory> cache = new ConcurrentHashMap<>(); // ← 多余
    private final ChatMemoryProvider provider;

    public ChatMemory get(String sessionId) {
        return cache.computeIfAbsent(sessionId,
            id -> provider.get(id)); // ← provider.get() 内部已经缓存了
    }
}
```

**正确做法**：

```java
// ✅ 正确：直接使用 Provider
public AIWorker createWorker(String sessionId) {
    ChatMemory memory = provider.get(sessionId); // 内部自动缓存
    return ...
}
```

### 陷阱 2：使用官方简单实现

**问题**：官方的 `MessageWindowChatMemory` 按消息数量清理，可能打断对话轮次。

**正确做法**：使用自定义的 `SafeTurnBasedChatMemory`（按轮次清理）。

### 陷阱 3：忘记清理过期记忆

**风险**：长期运行会导致内存泄漏。

**解决方案**：

```java
@Scheduled(fixedRate = 3600000) // 每小时
public void cleanupExpiredMemories() {
    // 清理超过 1 小时未使用的会话
    // 需要记录最后访问时间
}
```

## 🎯 最佳实践

### 1. 使用工厂模式

```java
// ✅ 推荐：工厂模式
@Component
public class AIWorkerFactory {
    public AIWorker createWorker(String sessionId) { ... }
}

// ❌ 不推荐：直接注入单例
@Autowired
private AIWorker aiWorker; // 所有用户共享
```

### 2. sessionId 命名规范

```java
// ✅ 推荐：有意义的 sessionId
String sessionId = "user:" + userId + ":" + conversationId;

// ❌ 不推荐：无意义的随机字符串
String sessionId = UUID.randomUUID().toString();
```

### 3. 记忆清理策略

```java
// 方案 A：用户主动清除
public void clearSession(String sessionId) {
    memoryProvider.clear(sessionId);
}

// 方案 B：定时清理过期
@Scheduled(fixedRate = 3600000)
public void cleanup() {
    // 清理 N 小时未访问的
}

// 方案 C：LRU 淘汰
// 使用带最大数量的缓存
```

### 4. 监控和日志

```java
@Component
public class AIWorkerFactory {

    public AIWorker createWorker(String sessionId) {
        log.debug("Creating AI worker for session: {}", sessionId);
        // ...

        // 监控活跃会话数
        log.info("Active sessions: {}",
            memoryProvider.activeSessionCount());
    }
}
```

## 📋 检查清单

在设计 AI 应用的聊天记忆时，确认以下事项：

- [ ] 不同用户/会话的记忆是否完全隔离？
- [ ] `ChatMemoryProvider` 内部是否有缓存机制？
- [ ] 是否避免了重复缓存设计？
- [ ] 是否有清理过期记忆的机制？
- [ ] 是否有监控活跃会话数的手段？
- [ ] sessionId 是否有清晰的命名规范？
- [ ] 是否使用了按轮次清理的记忆实现？

## 🔄 演进路径

### 阶段 1：单用户原型（开发初期）

```java
// 简单的单例 Agent
@Bean
public AIWorker aiWorker() {
    return AiServices.builder(AIWorker.class)
        .chatMemory(MessageWindowChatMemory.withMaxMessages(10))
        .build();
}
```

### 阶段 2：基础隔离（MVP）

```java
// 使用 Provider，但 Agent 还是单例
@Bean
public ChatMemoryProvider provider() {
    return memoryId -> MessageWindowChatMemory.builder()
        .id(memoryId)
        .maxMessages(10)
        .build();
}
```

### 阶段 3：完全隔离（生产就绪）

```java
// 工厂模式 + 自定义记忆实现
@Component
public class AIWorkerFactory {
    public AIWorker createWorker(String sessionId) {
        ChatMemory memory = safeMemoryProvider.get(sessionId);
        return AiServices.builder(AIWorker.class)
            .chatMemory(memory)
            .build();
    }
}
```
