## 一、核心概念

### 1.1 系统目标

模拟面试系统基于**岗位 JD + 用户简历**生成针对性面试问题，实现：
1. **深度挖掘**：针对简历中的项目经历和声称技能深入提问
2. **技能验证**：对比简历声称和实际回答水平
3. **差距分析**：指出简历与实际水平的"水分"

### 1.2 三 Agent 协同架构

```
┌─────────────────────────────────────────────────────────────┐
│                      面试流程                                │
├─────────────────────────────────────────────────────────────┤
│                                                              │
│  用户回答                                                     │
│     │                                                        │
│     ▼                                                        │
│  ┌────────────────┐    ┌────────────────┐    ┌──────────────┐│
│  │ 副面试官        │───▶│ 主面试官        │───▶│ 评审专家      ││
│  │ (选题控制)      │    │ (提问)          │    │ (评分)        ││
│  └────────────────┘    └────────────────┘    └──────────────┘│
│     决定下一个主题         生成问题              评估回答      │
│                                                              │
└─────────────────────────────────────────────────────────────┘
```

### 1.3 面试官提问策略

| 简历内容 | 提问示例 | 目的 |
|---------|---------|------|
| 项目经历 | "看到你做了电商秒杀项目，QPS 是多少？怎么解决的？" | 验证项目真实性 |
| 声称技能 | "你简历上写着精通 Redis，那 Redis 集群模式了解吗？" | 验证技能深度 |
| 技术栈 | "你项目中用了 Kafka，为什么选它而不是 RocketMQ？" | 考察技术选型能力 |
| 工作年限 | "3 年后端经验，那你说说 JVM 调优的实际案例？" | 评估经验匹配度 |

```sql
CREATE TABLE skill_tags (
    skill_id BIGINT PRIMARY KEY AUTO_INCREMENT,
    skill_name VARCHAR(50) NOT NULL UNIQUE,
    category VARCHAR(30),              -- LANGUAGE/FRAMEWORK/DATABASE/TOOL/ALGORITHM
    parent_id BIGINT,                  -- 支持层级：Java → Collection → HashMap
    difficulty_base INT DEFAULT 1,     -- 基础难度 1-5

```

### 2.3 模拟面试会话表 (mock_interview_sessions)

```sql
CREATE TABLE mock_interview_sessions (
    session_id VARCHAR(64) PRIMARY KEY,
    application_id BIGINT NOT NULL,
    user_id BIGINT NOT NULL,

    -- 快照（重要！避免后续更新影响历史记录）
    resume_snapshot JSON,
    jd_snapshot JSON,
    skills_snapshot JSON,

    -- 面试状态
    state VARCHAR(20),                 -- INIT/WELCOME/TECHNICAL_QA/HR_QA/FINISHED
    current_round INT DEFAULT 0,
    total_rounds INT DEFAULT 5,

    -- 面试进度
    skills_covered JSON,
    skills_pending JSON,

    -- 评分结果
    total_score DECIMAL(4,1),
    feedback_summary TEXT,
    resume_credibility_score DECIMAL(3,2)  -- 简历可信度评分
);
```

## 三、三 Agent 设计

### 3.1 主面试官 (MainInterviewerAgent)

**职责**：根据副面试官选定的主题，向用户提出面试问题

```java
public interface MainInterviewerAgent {
    String askQuestion(String context);
}
```

**System Message 关键点**：
- 提问策略：项目深挖 > 技能验证 > JD 要求覆盖 > 经验匹配度
- 严格约束：严禁评分、严禁重复、单一问题、长度控制
- 对话风格：专业但不失温和，带有"审视"态度

### 3.2 副面试官 (ViceInterviewerAgent)

**职责**：分析用户回答，决定下一个要考察的知识点

```java
public interface ViceInterviewerAgent {
    String decideNextStep(String context);
}
```

**输出格式** (严格 JSON)：
```json
{
  "action": "NEXT_QUESTION",
  "next_topic": "Redis 集群 Slot 迁移",
  "topic_source": "RESUME_SKILL",
  "reason": "简历上写着精通 Redis，需要验证深度",
  "difficulty": 4
}
```

**选题优先级**：
1. 项目深挖（最高优先级）
2. 技能验证
3. JD 要求覆盖

### 3.3 评审专家 (ExpertEvaluatorAgent)

**职责**：评估用户的回答质量，分析简历真实性

```java
public interface ExpertEvaluatorAgent {
    String evaluate(String context);
}
```

**评分维度**（总分 10 分）：
- 技术准确性（4 分）：概念是否正确，与简历声称的匹配度
- 逻辑清晰度（3 分）：表达是否条理清晰，符合 STAR 法则
- 深度与广度（3 分）：是否触及底层原理，与工作年限的匹配度

**输出格式** (严格 JSON)：
```json
{
  "scores": {"technical": 2.5, "logic": 2.0, "depth": 1.5},
  "total_score": 6.0,
  "credibility_assessment": {
    "match_level": "部分夸大",
    "gap_description": "简历声称精通 Redis，但对集群模式的回答很基础",
    "exaggeration_score": 0.6
  }
}
```

## 四、面试流程

### 4.1 完整流程

```
1. 用户发起模拟面试
   └─ POST /api/mock-interview/start
      └─ 创建 MockInterviewSession
      └─ 创建三 Agent 组
      └─ 进入 WELCOME 状态

2. 面试对话循环
   └─ POST /api/mock-interview/sessions/{id}/message
      ├─ 保存用户回答
      ├─ 副面试官决定下一个主题
      ├─ 主面试官生成问题
      ├─ 保存面试官问题
      └─ 进入下一轮

3. 评估回答
   └─ POST /api/mock-interview/sessions/{id}/evaluate
      ├─ 评审专家评估
      └─ 保存评分记录

4. 结束面试
   └─ POST /api/mock-interview/sessions/{id}/finish
      ├─ 生成面试总结
      ├─ 持久化所有 Agent 记忆
      └─ 更新会话状态为 FINISHED
```

### 4.2 状态流转

```
INIT → WELCOME → TECHNICAL_QA → HR_QA → GENERATING_REPORT → FINISHED
        │                                                │
        └───────────────── 可直接结束 ──────────────────┘
```

## 九、常见问题

### Q1: 为什么需要快照机制？

**A**: 确保面试时的数据不受后续修改影响。例如用户面试后更新了简历，不应该影响已完成的面试记录。

### Q2: 三个 Agent 如何协作？

**A**:
1. 副面试官分析用户回答，决定下一个考察主题
2. 主面试官根据选定主题，生成针对性的面试问题
3. 评审专家评估用户回答质量，分析简历真实性

### Q3: 技能标签为什么需要层级结构？

**A**:
```
Java (父技能)
  └─ Collection (子技能)
      └─ HashMap (具体知识点)
```
支持从粗到细的选题，先验证大类，再深入细节。
